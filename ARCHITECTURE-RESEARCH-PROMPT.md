# Промпт для нейронки: глубокий ресёрч АРХИТЕКТУРЫ F2P live-игры (вставь целиком)

Ты — ведущий backend/инфраструктурный архитектор F2P мобильных live-игр (выпускал тайтлы с миллионами DAU, проектировал бэкенды, анти-чит, LiveOps). Дай конкретный прикладной разбор как из ПРОТОТИПА выстроить АРХИТЕКТУРУ, способную дойти до прод-релиза и масштабироваться, для СОЛО-разработчика с минимальным бюджетом. Не общие слова — конкретные сервисы, схемы данных, миграционные пути, лимиты, цены, код-паттерны. Цитируй источники.

## КОНТЕКСТ ИГРЫ (текущая архитектура)
Киберпанк auto/idle-RPG на **Godot 4.7**, GDScript. Кросс-платформа: web (GitHub Pages) ✅ + Android-нативка (AAB собран) ✅, iOS планируется (через облачный CI).
- **Весь код в ОДНОМ файле `main.gd` (~5500 строк)** — монолит. Прогрессия, бой, UI, кланы, Firebase, локализация — всё вместе.
- **Бэкенд: Firebase** — Realtime Database (REST через HTTPRequest, test-mode ОТКРЫТ) + Anonymous Auth (через Identity Toolkit REST). Кланы: профили #ID, клан-боссы (общий HP), лидерборды, чат — всё прямые клиент→RTDB запросы.
- **Сейвы:** локальные (`user://save.json`, в web=IndexedDB). Облако-сейвов/линковки аккаунтов НЕТ.
- **Анти-чит:** НЕТ (клиент-авторитетный, RTDB открыт).
- **Аналитика:** нет замеренных KPI (планируется ByteBrew).
- **Прогрессия:** бесконечная (стадии до ~2344, престиж+сингулярность), числа float64.
- Соло-разработчик, бюджет минимальный, хочет монетизацию (IAP/реклама/гача).

## ЧТО РАЗОБРАТЬ (конкретно, с приоритетами для соло-дева)
1. **BACKEND-АРХИТЕКТУРА И МИГРАЦИЯ.** Firebase RTDB (текущий) — где упрётся (лимиты конкурентных соединений, цена Blaze, структура данных для лидербордов/кланов)? Когда и НА ЧТО мигрировать (Firestore? PlayFab? Nakama? свой бэкенд)? Дай миграционный путь по этапам DAU (100→10k→100k). Схемы данных для кланов/лидербордов что не деградируют.
2. **АНТИ-ЧИТ И БЕЗОПАСНОСТЬ (критично перед монетизацией).** Клиент-авторитетная idle-игра с открытым RTDB = читы тривиальны. Что РЕАЛЬНО защитить (покупки IAP — серверная валидация чеков Google/Apple; лидерборды/клан-вклад — серверная проверка; сейвы). Firebase Security Rules + App Check + Cloud Functions — конкретные правила/функции. Что НЕ стоит защищать для соло (прагматизм). Серверная валидация IAP — как (Cloud Functions + Google Play Developer API / App Store Server API).
3. **CLOUD-СЕЙВЫ + ЛИНКОВКА АККАУНТОВ.** Анон→Google/Apple линковка (не терять прогресс при смене устройства), конфликт-резолюция сейвов, античит сейвов (подпись/серверная валидация ключевых полей). Конкретная схема.
4. **LIVEOPS-ИНФРА (server-driven).** Remote Config (события/офферы/баланс БЕЗ апдейта приложения), A/B-тесты, feature-flags, расписание событий. Firebase Remote Config vs альтернативы. Как раздавать LiveOps-календарь + гача-баннеры server-side.
5. **АНАЛИТИКА-ПАЙПЛАЙН.** ByteBrew/GameAnalytics (готовые) vs свой (events→BigQuery). Какие события, как не терять при оффлайне, GDPR/consent. Связка с Remote Config для data-driven решений.
6. **CI/CD МУЛЬТИ-ПЛАТФОРМА.** GitHub Actions для Godot: web (Pages) + Android (AAB, подпись из secrets) + iOS (macOS-раннер, .ipa, signing). Конкретный workflow. Versioning, staged rollout.
7. **КОД-АРХИТЕКТУРА (рефакторинг монолита).** main.gd 5500 строк — когда и КАК разбивать (Godot-паттерны: autoload-синглтоны, ресурсы-данные, сцены-компоненты, сигналы)? Что выделить первым (бой/прогрессия/UI/нет-слой) без поломки. Стоит ли вообще для соло (риск vs выгода)? Тест-стратегия.
8. **СТОИМОСТЬ/МАСШТАБ.** Помесячная оценка инфры по этапам DAU (Firebase Blaze, Functions, хранилище, трафик). Где соло-дев попадёт на деньги. Бесплатные лимиты и когда кончатся.
