# Исследование: КЛАН-система и КЛАН-БОСС для cyber-auto-rpg

> Контекст: киберпанк mobile idle/auto-battler RPG на Godot 4, экспорт в HTML5 (GitHub Pages).
> Сейчас — ОДИНОЧНАЯ игра, сейвы в localStorage, бэкенда нет.
> Цель: добавить КЛАНЫ (группы друзей) + событие КЛАН-БОСС (общий босс с огромным HP, фиксированный урон от босса, вклад = сила игрока).
> Дата: 2026-06-28.

Документ построен по топикам: **находки → рекомендация**, затем общая секция **Firebase-схема** и **MVP-скоуп**.

---

## 0. TL;DR (выжимка решения)

- **Механика клан-босса:** копируем гибрид **RAID: Shadow Legends Clan Boss** + **AFK Arena Guild Hunting**. Один общий босс с гигантским HP-пулом, у каждого участника N «ключей» (попыток) в неделю. Каждая атака — это **детерминированная симуляция одного боя** на стороне клиента: игрок выживает X секунд против босса с **фиксированным уроном**, наносит урон по своей «силе», урон вычитается из общего HP. Награды — по тиру нанесённого урона (как сундуки RSL) + бонус топ-контрибьюторам + участие всем, кто ударил.
- **Расчёт силы:** идея владельца (**пиковый снапшот**) — правильная. `power = f(best_stage, sum_hero_levels, gear_score)`. Берём МАКСИМУМЫ за всё время (прокачка/стадия) + текущий гир — чтобы престиж не обнулял вклад. Это «Combat Power» как в MapleStory Idle / Idle Heroes — одно число.
- **Анти-чит, честный вердикт:** на **полностью клиентской** игре защититься от читера НЕЛЬЗЯ. Клиент сам считает урон → его можно подделать (localStorage, перехват запроса). Реалистично: **sanity-cap** (макс правдоподобный урон от заявленной силы) + **App Check** (отсечь не-наш-клиент/ботов) + **Cloud Function** как единственный путь записи урона (атомарный, с проверкой каппа и rate-limit). Это «достаточно для клана друзей», но не настоящий анти-чит. Решительный читер всё равно пробьёт — это нормально для казуальной соц-игры.
- **Firebase:** **Realtime Database (RTDB)**, не Firestore. Причина: дешёвый realtime-синк HP-бара на всех участников, атомарный `increment`/транзакция по одному узлу босса, простая tree-схема. Firestore словит document-contention при 20+ одновременных писателях в один документ.
- **Интеграция Godot↔Firebase:** **REST API через `HTTPRequest`** для записи (через Cloud Function HTTPS-endpoint) + **`JavaScriptBridge` к JS SDK** для realtime-подписки на HP-бар. Anonymous Auth даёт `uid` → маппим в `#ID`.
- **MVP:** создать/вступить в клан по коду, ОДИН недельный клан-босс, лидерборд вклада, награды за участие. Позже: роли, чат, несколько боссов, сезоны.

## 1. Структура КЛАНОВ/ГИЛЬДИЙ в idle/гача-играх

### Находки (реальные цифры)

**Размер клана** — в казуальных idle/гача обычно небольшой, под «обозримую группу»:
- Cookie Run: Kingdom — **30 участников** (1 лидер + 1 офицер).
- Idle Clans — **20 участников**.
- Idle Farm — до **50**.
- RAID: Shadow Legends — кланы до **30**.
- Tap Titans 2 — кланы до **50**, но активное ядро рейда меньше.
- AFK Arena гильдии — крупнее, но участие в Team Hunting ограничено по людям/попыткам.

**Роли / иерархия** (канон):
- **Leader (лидер)** — создатель. Меняет имя/описание клана, кикает, пишет анонсы, **стартует ивенты** (в нашем случае — клан-босса).
- **Officer (офицер)** — second-in-command, назначается лидером. Тоже может кикать и менять анонсы.
- **Member (участник)** — играет, вносит вклад.
- Спец-механика передачи власти: если лидер неактивен N дней (часто **14**), лидерство автоматически переходит офицеру; если оба молчат — самому свежеактивному участнику. **Это важно для клана друзей: иначе мёртвый лидер = мёртвый клан.**

**Join/leave/invite-флоу:**
- Вступление: по **коду/ID клана**, по приглашению, или из списка открытых кланов (с авто-аппрувом или заявкой).
- Для друзей оптимально: **публичный 6-значный код приглашения** — лидер шарит код в Telegram, друг вводит и сразу внутри.

**Прогрессия клана и что держит ретеншн:**
- AFK Arena: гильдия копит **Guild Activity Points** (донат активности), за порог (9000) **разблокирует второго босса Soren** — коллективная цель.
- Idle Heroes: **Guild Tech** — общий бафф всем героям всех участников за гильд-коины с рейда; **Guild Coins → Guild Store/Mill/Tech**. Это петля: бьёшь босса → коины → апаешь общий бафф → бьёшь сильнее.
- Cookie Run: ежедневный **check-in за награду** — простая привычка.
- Соц-обязательство: «раз ты в рейд-гильдии — тебя ждут, надо приходить» — главный драйвер удержания.

### Рекомендация для нас

- **Размер: 20–30 человек.** Для клана-друзей этого с запасом. Старт MVP — хватит и 20.
- **Роли в MVP — только Leader + Member.** Офицер и авто-передача лидерства — фаза 2 (но заложить поле `role` сразу).
- **Вступление — по 6-значному коду.** Лидер создаёт клан → получает код → кидает в Telegram → друзья вводят. Без модерации заявок на старте.
- **Прогрессия клана = клан-босс сам по себе** (общая цель «завалить босса на этой неделе») + лидерборд вклада. Гильд-валюту/гилд-теч — отложить, но это правильное направление развития ретеншна (петля «босс → валюта → общий бафф»).
- **Социальная привязка:** имя `#ID + ник` в лидерборде вклада — каждый видит, кто сколько вынес. Этого достаточно для соревновательного «не быть последним среди друзей».

## 2. КЛАН-БОСС / GUILD-BOSS: канон рейд-механики

### Находки по эталонам

**RAID: Shadow Legends Clan Boss (золотой стандарт):**
- **Ключи (keys):** игрок копит **1 ключ / 6 часов**, кап накопления — **2 ключа**. То есть ограниченное число попыток, нельзя бесконечно фармить.
- **Один общий босс на клан, общий HP-бар.** Каждый член клана бьёт — суммарный урон копится между всеми боями клана, в итоге босс падает.
- **Урон за бой = что ты выбил за один прогон команды** (бой детерминирован билдом). Можно потратить **несколько ключей, чтобы суммировать урон** и попасть в более высокий тир сундука.
- **Тиры наград (chests):** 4 уровня сундуков, какой дадут — зависит от **суммарного нанесённого тобой урона**. Завалить целиком один уровень босса (добить) → **всем атаковавшим двойные награды**.
- **Affinity-слой:** у босса меняется аффинити на 50% HP — добавляет тактики (нам не обязательно, но «фаза босса на 50% HP» — приятная деталь).
- Дизайн боя: «выжить как можно больше ходов и наносить стабильный DoT», а не бёрст — то есть **урон = функция выживаемости × dps**.

**AFK Arena Guild Hunting (Wrizz / Soren):**
- **Два босса**, Wrizz — ежедневный персистентный, Soren — разблокируется донатом активности гильдии.
- Игрок «наносит как можно больше урона за ограниченное время», **награда по % нанесённого урона**.
- **Энрейдж:** по мере падения HP Wrizz **повышает свой урон** (anti-stall — нельзя бесконечно тянуть).
- **Лимит попыток:** 2 раза за ивент (VIP6 — 3).
- Награды: **Guild Coins** (для гилд-стора) + **бонус топ-дамагеру**. Сундуки от Crude до Legendary + алмазы.

**Idle Heroes Guild Raid:**
- Награды **по рангу урона**: топ ~9000 коинов, низ ~5000. **Добивающий удар** даёт доп-награду «kill reward».

**Tap Titans 2 Clan Raid:**
- Босс из **частей с раздельным HP** (head/torso/руки/ноги) + слой брони у T2.
- **Каждый бьёт раз в 30 сек**, рейд = 3–8 Titan Lords.
- Урон скейлится от билда (clan ship, raid cards) — снова детерминированный «один тап = посчитанный урон».

### Сухой канон (что общего у всех)

1. **Общий HP-пул**, не персональные HP. Все бьют одного босса, урон суммируется.
2. **Огромный HP** (несколько часов/дней коллективного фарма) — это и есть «совместная цель».
3. **Ограниченные попытки** (ключи/N раз в день/кд) — иначе самый задротный игрок просто заспамит.
4. **Босс с фиксированным/детерминированным уроном** → твой урон = функция твоей силы и выживаемости, а не рандома. (Ровно идея владельца.)
5. **Anti-stall:** энрейдж или таймер боя, чтобы слабый не «зависал» вечно.
6. **Награды двухслойные:** (а) **тир/ранг по личному урону** (мотивация качаться) + (б) **участие всем, кто ударил** (никто не обижен) + (в) **бонус за добивание/топ**.

### Рекомендация: механика нашего клан-босса

**Модель «детерминированный одиночный заход» (RSL-style), адаптированная под наш auto-battler:**

- Лидер **стартует клан-босса** (раз в неделю). Создаётся объект босса с `hp_total` (огромный) и `endsAt` (+7 дней).
- У каждого участника — **N ключей в неделю** (старт: 3–5). Ключ = одна попытка.
- **Одна попытка = детерминированная симуляция боя:** игрок против босса, у босса **фиксированный DPS** (статичный урон/сек). Игрок наносит урон по своей силе, пока не умрёт (его эффективное HP / урон босса = время жизни) **или** до хард-капа времени боя (например 60 сек, anti-stall).
  - Итог попытки: `damage = effective_dps(power) × time_survived(power, boss_dps)`.
  - Слабо прокачанный → умирает за пару секунд, наносит мало. Сильный → живёт долго, наносит много. Ровно желаемое поведение.
- Урон **атомарно вычитается** из `hp_remaining` и **прибавляется** к `contributions[uid]`.
- Босс падает, когда `hp_remaining <= 0` → клану **clear-награда**, добивающему — бонус.
- Если за неделю не добили → раздаём награды по накопленному вкладу (participation), босс ресетится.

**Награды (двухслойно):**
- **По тиру личного вклада** (4 тира как RSL-сундуки: по % от `hp_total` или по абсолюту) — алмазы/гир/гача-валюта.
- **Участие:** ударил хоть раз → минимальный сундук.
- **Топ-3 по лидерборду вклада** → доп-бонус (мотивация качаться выше друзей).
- **Добивающий удар** → kill-бонус.

**Почему так, а не «реалтайм-тапанье одного босса всеми сразу»:** наш бэкенд — Firebase без авторитетного сервера. Реалтайм-кооп-бой (все одновременно дамажат, синк по тикам) потребовал бы серверной симуляции. «Детерминированный заход + атомарный вычет из HP» даёт **тот же эмоциональный эффект** (общий HP-бар тает на глазах у всех, см. realtime-подписку), но технически — это просто атомарные инкременты. Это реализуемо на RTDB без игрового сервера.

## 3. Расчёт СИЛЫ для честного вклада (prestige-proof)

### Находки

- В idle-RPG (MapleStory Idle, Idle Heroes) есть единый **Combat Power (CP)** — «одно число, сворачивающее все статы, чтобы сравнивать игроков». В CP входит ВСЁ: статы героев, HP/MP, перманентные апгрейды (Maple Rank), гир.
- Ключевой нюанс из MapleStory: CP — **оценка**, и фактический урон может не совпадать 1:1 с CP. Для нас это ок — нам нужна **монотонная, не-эксплойтабельная** функция, а не идеальная.
- Идея «пикового снапшота» = по сути собственный анти-абуз: берём не текущее (которое престиж обнуляет), а **исторический максимум**.

### Рекомендация: формула «пик-снапшота»

Хранить и обновлять (только вверх) per-player:
- `best_stage` — макс достигнутая стадия за всё время.
- `max_hero_levels` — сумма (или макс) уровней героев на пике.
- `gear_score` — скор текущего гира (гир переживает престиж → можно брать текущий).

Сводим в одно число `power`:

```
power = best_stage^A  ×  (1 + heroLevelFactor)  ×  (1 + gearScore/G)
```

где
- `best_stage^A` — главный драйвер (A≈1.2–1.5, чтобы глубина фарма доминировала, как item-level в Diablo, что уже заложено в проекте);
- `heroLevelFactor` = `sum_hero_levels / L` (нормировочный делитель);
- `gearScore` — сумма «уровней шмоток × тир-вес» (в проекте дроп уже падает с item-level = max(1,stage), это идеально стыкуется).

**Из `power` детерминированно выводим боевые величины для симуляции попытки:**
- `effective_dps = power × k_dps`
- `effective_hp  = power × k_hp`
- `time_survived = min(BATTLE_CAP, effective_hp / boss_dps)`
- `damage = effective_dps × time_survived`

**Почему именно так:**
- **Prestige-proof:** все «макс»-поля только растут (`max(old, new)`), престиж сбрасывает текущую стадию, но `best_stage` остаётся → вклад в клан не проседает. Ровно требование владельца.
- **Честность:** урон монотонно растёт по реальному прогрессу. Нельзя «вынести босса», не прокачавшись.
- **Анти-инфляция:** `power` пересчитывается ИЗ снапшота прогресса, а не задаётся напрямую. (Дальше — анти-чит, §4, ограничивает подделку самого снапшота.)

**Уточнение к идее владельца:** идея верная, добавить два штриха — (1) `power` должен быть **детерминированной функцией от снапшота**, который сам по себе валидируется/каппируется на сервере (не доверяем напрямую присланному `power`); (2) ввести **soft-cap/затухание** на очень больших стадиях, чтобы один сверх-задрот не превращал клан-босса в тривиальный (как энрейдж-механика anti-stall).

## 4. АНТИ-ЧИТ для клиент-авторитетной web-игры (честный разбор)

### Находки (жёсткая правда)

- Документированный факт: **«Я взломал Firebase-лидерборд за 5 минут»** — клиент пишет что хочет в свои writable-данные, **Security Rules в одиночку это не остановят**.
- Официально от Firebase: **App Check — НЕ анти-чит**. «Всё, кроме серверной валидации, даёт лидерборд, где топ — читеры».
- **Security Rules умеют:** проверять структуру/типы/диапазоны данных, что пишет только владелец своего узла (`auth.uid`), запрещать запись в чужие узлы, валидировать `newData` против простых констант и против `data` (старого значения). **НЕ умеют:** проверить, что присланное число «честно заработано» в игре — они не знают игровой логики и не видят историю боёв.
- **App Check (reCAPTCHA v3 для web):** аттестует, что запрос идёт от **твоего настоящего приложения**, а не от curl/бота/левого клиента. Режет массовый абуз и скрипты. Но **не мешает** легитимному клиенту, в котором человек поправил память/localStorage.
- **Cloud Functions:** единственное место, где можно вшить серверную проверку игровой логики до записи в БД.

### Что реально по слоям (от дешёвого к дорогому)

1. **Security Rules (обязательно, бесплатно):** урон пишется ТОЛЬКО через свой `contributions[auth.uid]`; `power`/снапшот — только свой узел; запрет писать `hp_remaining` напрямую (только через функцию/транзакцию с валидным дельта-знаком); типы и диапазоны.
2. **App Check + reCAPTCHA v3 (дёшево, включается в консоли):** отсекает не-наш-клиент, ботов, голый REST-флуд. Заметно поднимает порог для ленивого читера.
3. **Sanity-cap (главный практичный приём):** сервер знает `power` игрока → знает **максимально правдоподобный урон за одну попытку** = `effective_dps(power) × BATTLE_CAP`. Любая попытка с `damage > cap(power) × tolerance` — режется/обнуляется. Так подделка «999999999 урона при power=10» не проходит.
4. **Cloud Function как ЕДИНСТВЕННЫЙ путь записи урона (рекомендуется):** клиент НЕ пишет `damage` напрямую. Клиент шлёт «использую ключ» → функция сама пересчитывает урон из серверной копии `power`, проверяет наличие ключа (rate-limit/cooldown), атомарно вычитает из `hp_remaining`, пишет в `contributions`. Клиент не контролирует число.
5. **Валидация снапшота `power`:** `best_stage` можно расти **не более чем на разумную дельту** между апдейтами; ограничить частоту апдейтов; сверять с прогрессом. Полностью не защитит (см. ниже), но ловит грубую накрутку.
6. **Rate-limiting:** не больше N попыток в неделю (ключи) — даже подделанный клиент не заспамит, т.к. счётчик ключей серверный.

### Честный вердикт

**На полностью клиентской игре (Godot-web, вся логика на клиенте) НАСТОЯЩЕГО анти-чита нет.** Игрок физически исполняет код у себя → `power` и логику симуляции можно подменить. Даже Cloud Function доверяет тому `power`-снапшоту, который клиент когда-то записал; а снапшот формируется на клиенте.

Что **реально достижимо** и чего **достаточно для клана друзей**:
- Грубую накрутку («миллиард урона», curl-флуд, бот) — **ловим** (cap + App Check + серверный пересчёт + ключи).
- Умную накрутку (читер аккуратно завышает `best_stage` в пределах правдоподобия и медленно «прокачивает» фейк) — **не остановим** без авторитетного сервера, который сам симулирует геймплей.
- **Траст-модель для друзей:** этого хватает. Если кто-то из друзей читит — это видно по лидерборду (аномальный скачок), и решается социально (кикнуть). Не надо строить anti-cheat уровня соревновательного PvP для группы из 20 знакомых.

**Вывод:** ставим слои 1–4 + 6 (Rules + App Check + Cloud Function + sanity-cap + серверные ключи). Это «good enough for a friends casual clan». Полный анти-чит = переписать игру на серверную авторитетность, что вне скоупа.

## 5. FIREBASE-АРХИТЕКТУРА

### Выбор БД: **Realtime Database (RTDB)**, не Firestore

Обоснование:
- **Realtime-синк HP-бара:** RTDB даёт мгновенную подписку на изменение одного узла (`hp_remaining`) для всех участников — все видят, как босс тает. Это его коронная фича («10K-лидерборд, обновляется мгновенно»).
- **Атомарность по одному узлу:** урон по боссу — это инкремент `hp_remaining` и `contributions[uid]`. RTDB `transaction()` / REST `?x-firebase-...`/server-side `increment` на одном узле — просто и атомарно.
- **Firestore хуже здесь:** при 20+ одновременных писателях в ОДИН документ босса — document-contention, лаги 0.5–2с, ошибки. Это прямо про наш кейс (все бьют одного босса).
- Объёмы данных у нас крошечные (кланы друзей) → биллинг RTDB не проблема.

(Firestore оставить в уме на будущее, если понадобятся сложные запросы/сезонные истории — но для MVP RTDB.)

### Схема RTDB (tree / JSON)

```jsonc
{
  "players": {
    "$uid": {
      "nick": "NeonRider",
      "shortId": 1042,              // #1042 — человекочитаемый ID (счётчик)
      "clanId": "clan_ab12cd",
      "power": 18450,               // ПЕРЕСЧИТАННАЯ сила (пишет Cloud Function из снапшота)
      "snapshot": {                 // пиковый снапшот (валидируется)
        "bestStage": 312,
        "heroLevels": 240,
        "gearScore": 880,
        "updatedAt": 1719500000000
      },
      "lastSeen": 1719500000000
    }
  },

  "clans": {
    "$clanId": {                    // напр. "clan_ab12cd"
      "name": "Хромированные Псы",
      "joinCode": "X7K2Q9",         // 6-знач код для вступления
      "leader": "$uid",
      "createdAt": 1719000000000,
      "members": {                  // map, не array — дёшево добавлять/удалять
        "$uid": { "role": "leader", "joinedAt": 1719000000000 },
        "$uid2": { "role": "member", "joinedAt": 1719010000000 }
      },
      "activeBossId": "boss_2026w26"
    }
  },

  "joinCodes": {                    // обратный индекс код → clanId (для вступления)
    "X7K2Q9": "clan_ab12cd"
  },

  "clanBoss": {
    "$clanId": {
      "$bossId": {                  // "boss_2026w26" (год+неделя)
        "name": "ICE-Голем v9",
        "hpTotal": 50000000,
        "hpRemaining": 31200000,    // realtime-подписка ВСЕХ участников сюда
        "bossDps": 1200,            // фиксированный урон босса
        "battleCap": 60,            // anti-stall, сек
        "startedAt": 1719400000000,
        "endsAt": 1720004800000,
        "status": "active",         // active | cleared | expired
        "killedBy": null,
        "contributions": {          // лидерборд вклада
          "$uid":  { "nick": "NeonRider", "damage": 4200000, "keysUsed": 3 },
          "$uid2": { "nick": "GhostWire", "damage": 1100000, "keysUsed": 1 }
        }
      }
    }
  },

  "keys": {                         // серверный счётчик попыток (rate-limit)
    "$uid": {
      "$bossId": { "used": 3, "max": 5 }
    }
  },

  "counters": {
    "lastShortId": 1042             // транзакционный счётчик #ID
  }
}
```

### Как синкается состояние в реальном времени

- Каждый участник **подписан** (RTDB `on('value')` через JS SDK) на `clanBoss/$clanId/$bossId/hpRemaining` и на `.../contributions` → HP-бар и лидерборд обновляются у всех живьём.
- Атака **НЕ пишется клиентом напрямую**. Клиент вызывает **HTTPS Cloud Function** `attackBoss(clanId, bossId)`:
  1. Проверяет App Check токен + auth.
  2. Читает серверный `power` игрока и счётчик `keys`.
  3. Если ключ есть — детерминированно считает `damage = effective_dps(power) × min(battleCap, effHp/bossDps)`.
  4. **Транзакцией** уменьшает `hpRemaining`, увеличивает `contributions[uid].damage`, декрементит ключ.
  5. Если `hpRemaining <= 0` → `status=cleared`, `killedBy=uid`, начисляет награды.
- Так клиент управляет только «хочу ударить», а число урона рождается на сервере из доверенного снапшота → §4 sanity-cap встроен by design.

### Auth: Firebase Anonymous Auth

- При первом запуске — `signInAnonymously()` → стабильный `uid`.
- При создании профиля — транзакцией инкрементим `counters/lastShortId` → выдаём человекочитаемый `#shortId`. `uid` — внутренний ключ, `#shortId + nick` — то, что видят друзья.
- Гость переживает перезагрузки (токен в IndexedDB). Минус анонимки: смена устройства/чистка браузера = новый uid (потеря привязки) — на будущее можно линковать к Google/email, но для MVP анонимка ок.

### Интеграция Godot-web ↔ Firebase

Два пути, **рекомендуется гибрид**:

**A. `JavaScriptBridge` → Firebase JS SDK (для realtime-подписок) — основной для чтения.**
- В `index.html` (наш экспортный шаблон в `docs/play/`) подключаем Firebase JS SDK, инициализируем app, делаем `signInAnonymously`, вешаем `onValue` на узлы босса, и при изменении — кладём JSON в глобал/вызываем колбэк, который Godot читает через `JavaScriptBridge.get_interface()` / `eval` / callback-объект (`JavaScriptBridge.create_callback`).
- Плюс: настоящий realtime-push (HP-бар живой), SDK сам рулит **рефрешем auth-токена**, реконнектом, оффлайн-кэшем.
- Это правильный путь именно для **подписки** на меняющийся HP.

**B. `HTTPRequest` → REST (для записи через Cloud Function) — основной для действий.**
- Действия (`createClan`, `joinClan`, `attackBoss`, `updateSnapshot`) — это вызовы **HTTPS Cloud Functions** обычным `HTTPRequest.request(url, headers, METHOD_POST, body)`.
- В заголовке — `Authorization: Bearer <idToken>` (берём idToken из JS SDK через bridge) + App Check токен.
- Можно и напрямую в RTDB REST (`https://<db>.firebaseio.com/path.json?auth=<idToken>`), но для записи урона — только через функцию (§4).

**Гочи (подводные камни):**
- **Auth-токен живёт ~1 час** → надо рефрешить. Если читаем idToken через JS SDK — SDK рефрешит сам; если гоняем чисто по REST — придётся вручную дёргать `securetoken.googleapis.com` рефреш. → ещё причина держать JS SDK для auth.
- **CORS:** RTDB REST CORS-дружелюбен; Cloud Functions (HTTPS onRequest) — нужно явно выставить CORS-заголовки (или использовать `onCall` + SDK через bridge). С GitHub Pages (другой origin) это критично — проверить.
- **App Check на web** = reCAPTCHA v3, нужен site key, и enforcement включается в консоли (сначала мониторинг, потом enforce, чтобы не отрезать живых игроков).
- **Security Rules** должны запрещать прямую запись `hpRemaining`/`contributions` обычным клиентам — только Admin SDK (Cloud Function) пишет туда; игроки пишут лишь свой `players/$uid` и читают босса.
- Готовый плагин **GodotNuts/GodotFirebase** (GDScript) существует, но он заточен под нативные REST-флоу; для **web**-экспорта realtime-подписку всё равно надёжнее делать через JS SDK + bridge. Можно подсмотреть в нём REST-обвязку auth.

## 6. MVP-СКОУП

### MVP (первый шип) — минимально жизнеспособный клан-босс

**Бэкенд:**
- Firebase проект: RTDB + Anonymous Auth + App Check (reCAPTCHA v3, режим monitor на старте).
- 3–4 Cloud Functions: `ensureProfile` (выдать #ID), `createClan`, `joinClan(code)`, `attackBoss(clanId,bossId)`.
- Security Rules: игрок пишет только свой `players/$uid`; читает свой клан и босса; запись босса/вклада — только Admin SDK.

**Фичи:**
1. **Профиль:** анонимный вход → `#ID + ник` (ник задаётся один раз).
2. **Клан по коду:** создать клан (→ 6-знач код) / вступить по коду. Лимит 20 чел. Роли: только leader+member.
3. **Один недельный клан-босс:** лидер жмёт «Старт босса» → общий HP + `endsAt` (+7д). У каждого 3–5 ключей.
4. **Атака:** кнопка «Ударить» (тратит ключ) → Cloud Function считает урон из снапшота `power` → HP падает.
5. **Realtime HP-бар + лидерборд вклада** (подписка через JS SDK): видно, как босс тает и кто сколько вынес.
6. **Снапшот силы:** клиент шлёт `{bestStage, heroLevels, gearScore}` (макс-значения) → функция пересчитывает и кэширует `power` (с sanity-валидацией дельты).
7. **Награды:** при добивании или по истечении недели — тир по личному вкладу + участие всем, кто ударил. (В MVP можно начислять простое — алмазы/гир-токены в существующую систему.)

**Анти-чит в MVP:** Rules + App Check(monitor) + урон считает функция из серверного `power` + ключи серверные + sanity-cap. Этого достаточно для друзей.

### Фаза 2 (после того как первый босс заходит)

- **Роли:** officer, права кика/анонса, **авто-передача лидерства** при неактивности 14д.
- **Гилд-валюта + гилд-теч** (петля ретеншна Idle Heroes): за вклад → коины → общий пассивный бафф клану.
- **Несколько боссов / тиры сложности** (Easy→Nightmare как RSL), второй босс по донату активности (как Soren AFK).
- **Фаза босса** (смена «аффинити»/механики на 50% HP).
- **Чат клана** (или хотя бы лента событий: «X добил босса!»).
- **App Check enforce** (после стабилизации).
- **Линковка аккаунта** (Google/email) чтобы не терять прогресс при смене устройства.

### Фаза 3 (соц-/лайв-сервис)

- **Сезоны** клан-боссов с ладдером кланов, награды за топ-клан.
- **Клан-войны** (клан vs клан по суммарному урону).
- Кросс-клан лидерборды, миграция холодных данных в Firestore для аналитики/истории.
